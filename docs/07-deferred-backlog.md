# 보류 백로그

이 문서는 지금 당장 구현하지 않는 확장 후보와 **다시 검토할 조건**을 기록한다.

## 보류 기준

| 기준 | 설명 |
| --- | --- |
| 실제 마찰 없음 | 아직 사용자가 반복해서 겪은 문제가 아니다. |
| 범위 확장 큼 | 정책 레이어를 런타임/플랫폼으로 바꿀 수 있다. |
| 유지비 큼 | 도입 후 버전, 문서, 테스트를 계속 관리해야 한다. |
| 더 작은 대안 있음 | 지침, 문서, doctor, CI로 충분히 대응 가능하다. |

## 보류 항목

| 항목 | 왜 보류 | 재검토 조건 |
| --- | --- | --- |
| `pilotrails init` / `pack` CLI | 수동 복사 도입의 반복 마찰이 아직 충분히 관찰되지 않았다. | 기존 `.github/` 충돌, 누락 파일, 설치 실수 이슈가 반복될 때 |
| MCP / Playwright 통합 | 자체 실행 플랫폼으로 범위가 커질 수 있다. | 외부 도구/브라우저 검증이 실제 작업에서 반복 필요해질 때 |
| 텔레메트리 / run log | 추적할 운영 데이터가 아직 충분하지 않다. | 여러 dogfood 세션의 시간, 실패, 검증 결과를 비교해야 할 때 |
| 플러그인 / manifest | 현재 자산 수는 수작업 배선과 doctor로 관리 가능하다. | agent, hook, skill 수가 늘어 배선 누락이 반복될 때 |
| eval 실행기 | 평가할 PilotRails 변형이 아직 많지 않다. | 여러 구성의 결과를 재현 가능하게 비교해야 할 때 |
| devcontainer / Docker | Node 기반 검사만으로 현재 재현성이 충분하다. | 환경 차이로 검증 실패가 반복될 때 |

## 설계상 비채택

| 항목 | 이유 |
| --- | --- |
| 상시 멀티에이전트 코딩 | 코드 작성 책임과 컨텍스트 연속성이 흐려진다. Explore는 read-only 탐색으로 제한한다. |
| 작업 도중 모델 전환 | 검증과 책임 추적이 어려워진다. 모델/모드 전환은 작업 경계에서만 한다. |

보류 항목을 채택하게 되면 [05-decision-log.md](05-decision-log.md)에 결정과 근거를 옮겨 적는다.
