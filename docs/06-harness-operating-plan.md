# 운영 원칙

PilotRails는 기능을 많이 쌓는 프로젝트가 아니라, Copilot 기반 작업을 **예측 가능하고 검증 가능하게 만드는 정책 레이어**다. 운영 원칙은 단순하다.

## 1. 관찰된 실패에서만 강화한다

새 hook, 새 스크립트, 새 템플릿은 실제 작업에서 반복 마찰이 관찰될 때 추가한다. 추측성 기능은 바로 구현하지 않고 [07-deferred-backlog.md](07-deferred-backlog.md)에 보류 조건과 함께 기록한다.

## 2. 완료는 검증 결과로 말한다

완료 보고에는 최소한 다음 중 관련 항목을 포함한다.

- 실행한 테스트/검사 명령
- 통과/실패 결과
- 기존 실패와 신규 실패의 구분
- 검증하지 못한 영역

## 3. 보호 경로는 작게 유지한다

보호 경로가 많아지면 생산성이 떨어진다. 현재 보호 대상은 상태 정본, always-on 지침, 결정 로그, hook 스크립트다. 새 보호 경로는 실제 오편집이 반복될 때만 추가한다.

## 4. 문서와 구현은 함께 움직인다

PilotRails는 문서형 정책과 코드형 검사가 함께 작동한다. 따라서 다음 변경은 항상 같이 검토한다.

| 변경 | 함께 확인할 것 |
| --- | --- |
| hook 추가/수정 | hooks.json, tests, docs/02 |
| 보호 경로 변경 | protect-paths, docs/02, harness-doctor |
| 새 agent/skill 추가 | README, feature_list, doctor 참조 |
| 완료 규칙 변경 | /finish, verify-done, CONTRIBUTING |

## 5. 공개 저장소로 운영한다

공개 사용자를 위해 다음 기준을 유지한다.

- README는 현재 기능과 사용법을 먼저 설명한다.
- 배경 리서치는 아카이브로 분리하고 최신 랭킹처럼 보이지 않게 한다.
- SECURITY, CONTRIBUTING, CODEOWNERS, 이슈/PR 템플릿을 유지한다.
- CI는 최소한 doctor, tests, smoke를 계속 실행한다.

## 6. 재평가 주기

큰 구조 변경 전에는 다음 순서로 확인한다.

1. 현재 실패가 실제로 반복됐는가?
2. 기존 지침이나 테스트로 해결할 수 없는가?
3. 새 장치가 추가 유지비보다 큰 이득을 주는가?
4. 제거하거나 보류할 수 있는 기존 장치는 없는가?
