# 설계 결정 근거

이 문서는 PilotRails의 주요 설계 결정과 채택 기준을 기록한다.

## 1. 결정 기준

| 기준 | 설명 |
| --- | --- |
| Copilot-native | VS Code Copilot이 인식하는 표준 커스터마이징 경로를 우선한다. |
| 최소 권한 | 작업 단계에 필요한 도구만 허용한다. |
| 검증 우선 | 완료 판단은 실제 명령 결과에 둔다. |
| 단일 쓰기 책임 | 코드 작성은 Build가 책임지고, 서브에이전트는 read-only로 제한한다. |
| 관찰된 실패 | 반복 관찰된 마찰에만 새 장치를 추가한다. |

## 2. 채택한 결정

| 결정 | 이유 |
| --- | --- |
| Plan / Build / Ask 분리 | 탐색과 편집을 섞지 않아 의도치 않은 변경을 줄인다. |
| Explore read-only 서브에이전트 | 넓은 탐색을 메인 컨텍스트 밖으로 격리하되 쓰기 책임은 분산하지 않는다. |
| 단일 always-on 지침 | 항상 주입되는 컨텍스트를 작게 유지하고 중복 규칙을 피한다. |
| Agent hooks | 보호 경로와 완료 게이트처럼 반복 가능한 규칙은 기계가 검사하게 한다. |
| CI 보완 | 로컬 hooks 비활성 환경에서도 최소 거버넌스 검사를 유지한다. |

## 3. 비채택 또는 제한한 결정

| 결정 | 현재 입장 |
| --- | --- |
| 자체 런타임 | 이 프로젝트의 범위 밖이다. |
| 상시 멀티에이전트 코딩 | 컨텍스트와 책임이 분산되므로 read-only 탐색에 한정한다. |
| 작업 중 모델 전환 | 작업 경계에서만 모델·모드를 바꾼다. |
| 샌드박스 실행 플랫폼 | 반복 마찰이 관찰되기 전까지는 보류한다. |
| 텔레메트리/플러그인 시스템 | 운영 데이터가 충분히 쌓인 뒤 재검토한다. |

## 4. 상충을 다루는 방식

PilotRails 기능은 서로 충돌할 수 있다. 예를 들어 자율 실행은 빠르지만 승인 게이트와 충돌하고, 세밀한 로그는 관측성을 높이지만 컨텍스트와 유지비를 늘린다.

충돌 시 우선순위는 다음과 같다.

1. 사용자 안전과 비가역 작업 승인
2. 검증 가능한 완료
3. 컨텍스트 연속성
4. 속도와 자동화

이 원칙에 따라, 편의 기능이 검증·안전·연속성을 해치면 채택하지 않는다.
